# sci-plan-nexus — Technical Architecture

A full-stack web app that converts a one-sentence scientific hypothesis into a fully-cited, schema-validated experiment plan (literature, overview, protocol, materials, budget, timeline, validation) via a multi-agent LLM pipeline.

## 4. Frontend

### 4.1 Stage Machine

[`routes/index.tsx`](src/routes/index.tsx) holds three stages:

1. **`input`** — empty desk, central notebook with a `<textarea>`. On submit, calls the `runQC` server function (novelty-only, ~1 s) so the literature modal can open immediately.
2. **`paper-only`** — only the “Paper” item is on the desk, modal opens with the novelty signal + references. The user clicks **Generate Full Plan**.
3. **`full-desk`** — `streamPlan(hypothesis)` opens an SSE connection. As each section completes, the corresponding desk item drops in. When `done` fires, the validated `Plan` is set in state and all five modals can be opened.

### 4.2 Live Progress UX

The page maintains two sets — `running: Set<string>` and `done: string[]` — driven by SSE events (`agent_start`, `section`, `info`). The status label reflects **real concurrency**:

- 1 running → `Running protocol agent…`
- 2+ running → `Running 3 agents in parallel: protocol, materials, validation…`
- 0 running → `✓ <last section> complete`

This is not faked — the labels are wired to the same events the server emits.

### 4.3 SSE Consumer

[`src/lib/plan-stream.ts`](src/lib/plan-stream.ts) is a hand-rolled SSE reader (no `EventSource`, because we POST a JSON body). It:

1. Reads `res.body` as a `ReadableStream<Uint8Array>`.
2. Splits on `\n\n` frame boundaries.
3. Parses every `data:` line as JSON.
4. Routes `{type:"event"}` → `onEvent`, `{type:"done"}` → resolve, `{type:"error"}` → throw.

### 4.4 Design System

- Tailwind v4 with custom **signal-** color tokens (cyan, violet, mint, amber, rose) used throughout the desk metaphor.
- `font-display` italic + monospace (`mono` utility) creates the “lab notebook” feel.
- Modal scrolling fix: [`desk-modal.tsx`](src/components/desk-modal.tsx) uses `flex flex-col` with `flex-1 min-h-0 overflow-y-auto` body, so long protocols / budgets scroll inside the modal independently of the footer.
- Timeline gantt is wrapped in `overflow-x-auto` with a per-week `min-width`, so 20-week plans remain readable.

## 5. Backend

### 5.1 Two Endpoints

| Endpoint              | Type                | Purpose                                              |
| --------------------- | ------------------- | ---------------------------------------------------- |
| `runQC` server-fn     | TanStack server-fn  | Fast novelty pass — paper modal opens in ~1 s        |
| `POST /api/plan`      | SSE route handler   | Full pipeline, streams progress + final plan         |

Both validate input with the same Zod shape (`hypothesis: z.string().min(10).max(2000)`).

### 5.2 SSE Wire Format

`POST /api/plan` returns `text/event-stream`. Each frame is a single `data: <json>\n\n` of one of three messages:

```ts
{ type: "event"; event: SectionEvent }   // many of these
{ type: "done";  plan: Plan }            // exactly one
{ type: "error"; message: string }       // only on failure
```

Where `SectionEvent` is:

```ts
type SectionEvent =
  | { kind: "agent_start"; section: string }
  | { kind: "token";       section: string; delta: string }
  | { kind: "section";     section: string; content: unknown; latency_ms: number }
  | { kind: "info";        message: string }
  | { kind: "warning";     message: string };
```

The `controller.close()` and `controller.enqueue()` calls are wrapped in try/catch so a client disconnect doesn’t crash the worker.

CORS is wide open (`*`) — fine for a hackathon, lock down in production.

## 6. Orchestrator — The Heart of the System

[`src/server/orchestrator.ts`](src/server/orchestrator.ts) runs the pipeline in **6 phases**:

### Phase 1 — Classify
`classifyHypothesis()` ([`retrieval/classify.ts`](src/server/retrieval/classify.ts)):
- Heuristic first (keyword buckets — instant, free, covers the 4 demo samples).
- LLM fallback constrained to a Zod enum if the heuristic returns `"general"`.
- Returns `{ domain, experiment_type }` used for retrieval profile selection and corrections lookup keys.

### Phase 2 — Retrieve (parallel)
`Promise.all` of three Tavily searches: `protocols`, `suppliers`, `papers`.
- Live backend in [`retrieval/tavily.ts`](src/server/retrieval/tavily.ts) with **in-memory SHA-256 cache** (identical hypothesis = 0 credits) and a budget of 3 calls × 1 credit per plan.
- [`retrieval/index.ts`](src/server/retrieval/index.ts) auto-selects: Tavily if `TAVILY_API_KEY` is set, else `mockRetrieval`.

### Phase 3 — Corrections lookup (parallel, opt-in)
For each of the 6 sections, ask the corrections store for the top-K reviewer corrections matching `(experiment_type, section)`. Only fired when `applyCorrections=true` (set by a stretch-goal `/api/regenerate` endpoint). The mock store ([`corrections/mock.ts`](src/server/corrections/mock.ts)) seeds two example corrections; a real impl would be pgvector-backed.

### Phase 4 — Stage 1 fan-out (5 agents in parallel)
```
novelty │ overview │ protocol │ materials │ validation
```
Each agent gets:
- The hypothesis, domain, experiment type.
- Its slice of retrieval (`protocols`, `suppliers`, `papers`).
- Its filtered corrections (only relevant section).
- An optional `onDelta` token sink (currently emits `kind:"token"` events).

Agents emit `agent_start` before they run and a `section` event with `latency_ms` when they complete.

### Phase 5 — Stage 2 (dependent)
- **Timeline** runs after **protocol** (it needs the step list to phase the gantt).
- **Budget** runs after **materials + timeline** (needs reagent costs and labor weeks).

### Phase 6 — Repair → Lead PI → Schema gate

1. **`groundingRepair`** — deterministic, free, runs *before* the LLM-style consistency pass:
   - Protocol steps without an `http(s)` citation → inject the first available URL from `protocols/papers/suppliers`.
   - Missing `budget.reagents` line while materials non-empty → synthesize one covering `materials.total_usd`, cited from a material’s `catalog_url`.
   - Budget lines without a citation → inject a category-appropriate URL (suppliers for reagents/consumables, BLS for labor/overhead, etc.).
   - This single function saves the demo when LLMs forget to paste URLs.
2. **`consistencyPass`** ([`agents/leadPI.ts`](src/server/agents/leadPI.ts)) — see §8.
3. **`PlanSchema.parse(plan)`** — final hard gate; throws if anything is still non-conformant.

## 7. Agent Contract

Every agent is a pure async function that:
1. Builds a system + user prompt (helpers in [`agents/base.ts`](src/server/agents/base.ts)):
   - `COMMON_RULES` — “JSON only”, “cite real URLs from the retrieval block”, etc.
   - `formatRefs(items)` — pretty-prints retrieval results as `[1] title — url`.
   - `formatCorrections(cs)` — pretty-prints prior reviewer corrections as `BEFORE / AFTER / WHY`.
2. Calls `generateJSON({ schema: <PerSectionSchema> })`.
3. Returns `{ data, trace }`.

All section schemas are **derived from `PlanSchema.shape.<section>`**, so there is no drift between an agent’s output and the final plan’s expected shape.

Example (protocol):

```ts
return generateJSON({
  agent: "protocol",
  system,
  user,
  schema: ProtocolSchema,        // = PlanSchema.shape.protocol
  onDelta: ctx.onDelta,
});
```

## 8. Lead PI — Deterministic Consistency Pass

[`agents/leadPI.ts`](src/server/agents/leadPI.ts) is *not* an LLM agent — it’s a TS function that walks the assembled `Plan` and applies cross-section invariants. It returns `{ plan, report: { fixed[], warnings[] } }`, throwing `LeadPIRejection` only on hard violations.

| # | Rule                                                           | Action          |
| - | -------------------------------------------------------------- | --------------- |
| 1 | `materials.total_usd` = Σ items.total_cost_usd                | auto-recompute  |
| 2 | `budget.total_usd`    = Σ lines.cost_usd                       | auto-recompute  |
| 3 | Missing `budget.reagents` line when materials non-empty        | **violation**   |
| 3b| `reagents` line < materials total                              | bump + re-sum   |
| 4 | `timeline.weeks` = max(start + duration) of phases             | auto-recompute  |
| 5 | `timeline.weeks > 0` and `phases.length ≥ protocol.steps`     | **violation**   |
| 6 | `phase.depends_on` references existing phase IDs              | warning         |
| 7 | `critical_path` IDs exist                                      | warning         |
| 8 | Every protocol step has ≥ 1 valid `http(s)` citation           | **violation**   |
| 9 | Every budget line has ≥ 1 valid `http(s)` citation             | **violation**   |
| 10| Every material has a valid `catalog_url`                       | **violation**   |

Each `fixed` and `warning` is also emitted as an `info` / `warning` SSE event so the user sees “lead-PI fixed: budget.total_usd recomputed to $4,231” live.

## 9. LLM Client

[`src/server/llm/client.ts`](src/server/llm/client.ts) is the single entry point for every model call. Public API:

```ts
generateJSON<T extends ZodTypeAny>({
  agent, system, user, schema,
  stubKey?, temperature?, onDelta?,
}): Promise<{ data: z.infer<T>, trace: AgentTrace }>;
```

### 9.1 Provider Resolution

| `LLM_PROVIDER` env value | Backend                                                       |
| ------------------------ | ------------------------------------------------------------- |
| `groq` (default)         | OpenAI SDK, `baseURL=https://api.groq.com/openai/v1`          |
| `openai`                 | Native OpenAI SDK                                             |
| `gemini`                 | `@google/genai`, `responseMimeType=application/json` + schema |
| `stub`  (or `STUB_LLM=1`)| Deterministic fixtures from [`llm/stub.ts`](src/server/llm/stub.ts)            |

Default model env: `GROQ_MODEL=llama-3.3-70b-versatile`, `GEMINI_MODEL=gemini-2.5-flash`, `OPENAI_MODEL=gpt-5.2`.

### 9.2 JSON Mode + Repair Loop

For every provider:

1. First call with `temperature=0.2` and JSON-mode (`response_format` for OpenAI/Groq, `responseSchema` for Gemini).
2. Try `safeJSON(raw)` → `schema.safeParse(json)`.
3. If parse fails, send a **repair message** appending the previous output and the Zod issues:
   ```
   - /materials/items/0/catalog_url: invalid url
   ```
   with `temperature=0`. Re-parse.
4. If still failing, throw — caller will surface it as an SSE `error` event.

Groq specifically also has a fallback: if the model returns a sporadic `"Failed to generate JSON"` 400, retry **without** JSON mode at temperature 0.

### 9.3 Rate-Limit + Retry Wrapper

Every provider call goes through `withRateLimit()` ([`llm/ratelimit.ts`](src/server/llm/ratelimit.ts)):

- Global concurrency cap (`LLM_CONCURRENCY`, default 3) — protects free-tier quotas during the 5-way fan-out.
- Min interval between starts (`LLM_MIN_INTERVAL_MS`, default 0).
- Exponential-backoff retry (`LLM_MAX_RETRIES`, default 5) on transient errors.

### 9.4 Stub Provider

Setting `STUB_LLM=1` (or `LLM_PROVIDER=stub`) makes `generateJSON` return deterministic fixtures keyed by `agent`. This lets the entire pipeline run **offline** — useful for CI, frontend dev, and screenshots.

## 10. Retrieval Layer

### 10.1 Contract

```ts
interface RetrievalClient {
  protocols(hypothesis: string): Promise<RetrievalResult[]>;
  suppliers(hypothesis: string): Promise<RetrievalResult[]>;
  papers(hypothesis: string):    Promise<RetrievalResult[]>;
}

interface RetrievalResult { title: string; url: string; snippet?: string; }
```

The orchestrator never imports a concrete client — it uses `args.retrieval ?? mockRetrieval`, dependency-injected from `pickRetrieval()`.

### 10.2 Tavily Backend

[`retrieval/tavily.ts`](src/server/retrieval/tavily.ts):
- `search_depth: "basic"` (1 credit/call).
- `max_results: 3` per query.
- 3 queries per plan → ~333 plans worst-case on a 1000-credit dev key.
- In-memory `Map<sha256(query+kind), CacheEntry>` — same hypothesis = 0 extra credits.
- Per-domain query *profiles* live in [`retrieval/profiles.ts`](src/server/retrieval/profiles.ts), tuned per `(domain, kind)` pair (e.g. perovskite → protocols.io + Nature Energy, biosensor → ACS Sensors, etc.).

### 10.3 Mock Backend

`mockRetrieval` ([`retrieval/mock.ts`](src/server/retrieval/mock.ts)) returns curated lists per keyword. Used in CI, offline dev, and as the safety net when `TAVILY_API_KEY` is absent.

A separate validator ([`retrieval/validateCatalog.ts`](src/server/retrieval/validateCatalog.ts)) sanity-checks supplier URLs.

## 11. Shared Schema — Single Source of Truth

[`src/server/shared/types.ts`](src/server/shared/types.ts) defines **one** `PlanSchema` (Zod). Highlights:

- `Reference` / `Citation` — both require `z.string().url()`.
- `ProtocolStep` — `n` (positive int), `duration_hours` (≥ 0), and `citations.min(1)`.
- `Reagent` — `qty: z.coerce.string()` so models can return `100` *or* `"100 mg"` without breaking.
- `BudgetLine.category` — strict enum: `reagents | consumables | equipment | labor | overhead | other`.
- `TimelinePhase.depends_on` — array of phase IDs (validated by leadPI rule #6).
- `Metric` — name + threshold + method, all required.
- `provenance.agent_traces[]` — captures `{ agent, model, latency_ms, tokens_in?, tokens_out? }` for each agent so the UI can show per-section timings.

Per-section schemas (`OverviewSchema`, `ProtocolSchema`, …) are re-exported from `PlanSchema.shape.*` so agent outputs and final plan validation **cannot drift**.

## 14. Failure-Mode Map

| Failure                                          | Where it’s caught                                       |
| ------------------------------------------------ | ------------------------------------------------------- |
| Bad request body                                 | Zod `Body.parse` in `routes/api/plan.ts` → 400          |
| Provider 4xx/5xx (transient)                     | `withRateLimit` retry                                   |
| Groq sporadic “Failed to generate JSON” 400      | `client.ts` retries without JSON mode                   |
| LLM returns non-JSON                             | `safeJSON` + repair pass                                |
| LLM returns wrong shape                          | Zod issues fed back into a repair message               |
| LLM forgets a citation URL                       | `groundingRepair` injects from retrieval block          |
| LLM under-reports `timeline.weeks`               | `consistencyPass` recomputes from phases                |
| Materials sum ≠ declared total                   | `consistencyPass` recomputes                            |
| Missing `budget.reagents`                        | `consistencyPass` throws `LeadPIRejection`              |
| Final plan still invalid                         | `PlanSchema.parse()` throws — surfaced as SSE `error`   |
| Client disconnect during stream                  | `controller.enqueue` wrapped in try/catch               |
| Frontend receives `error` SSE                    | `streamPlan` rejects → caught in `index.tsx` `catch`    |

## 16. Why It’s Built This Way

- **Schema-first.** A single Zod source means agents, repair logic, and UI all speak the same language. Drift is impossible.
- **Determinism wraps the LLM.** Two layers (`groundingRepair` + `consistencyPass`) mean the system can ship a valid plan even when an agent misbehaves. The LLM is a powerful suggestion engine, not the source of truth.
- **Streaming first.** Every section is observable as it lands — the user sees real concurrency rather than a 10-second spinner.
- **Provider-agnostic.** Groq / OpenAI / Gemini / stub all share one entrypoint; no agent code knows or cares which model it’s talking to.
- **Cheap by default.** Heuristic classifier, basic-depth Tavily, in-memory cache, stub mode for CI — the system is designed to demo on a free tier.
